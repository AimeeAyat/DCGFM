Think of GIN as:
“Each node repeatedly mixes its own info with its neighbors, then the whole graph is summarized into one fingerprint.”

1. GCN (Graph Convolutional Network)
“Each node becomes the average of its neighbors”

Problem:
Different structures can collapse to same representation
Not very expressive

2. GraphSAGE
“Node keeps its identity + learns from neighbors”

Advantage over GCN:
Doesn’t mix self + neighbors blindly

3. GIN (Graph Isomorphism Network)

What it does:
Uses SUM aggregation (not mean)
Adds weighted self-feature
Passes through MLP (not just linear)
Key properties:
Most expressive
Can distinguish graph structures (WL test power)
Learns structure differences precisely
Intuition:

“Exact counting + powerful transformation”

| Feature               | GCN                  | GraphSAGE         | GIN                    |
| --------------------- | -------------------- | ----------------- | ---------------------- |
| Aggregation           | Mean (normalized)    | Mean / Max / LSTM | **Sum**                |
| Self handling         | Mixed with neighbors | **Concatenated**  | Weighted (1+ε)         |
| Transformation        | Linear               | Linear            | **MLP (nonlinear)**    |
| Expressiveness        | Low                  | Medium            | **High (WL powerful)** |
| Structure sensitivity | Weak                 | Medium            | **Strong**             |


CRITICAL DIFFERENCE
Mean vs Sum
GCN / GraphSAGE:
[1,1,1] → mean = 1  
[2,2]   → mean = 2  

[1,1,1] vs [1,1]
→ both mean = 1 ❌ indistinguishable

GIN
sum([1,1,1]) = 3  
sum([1,1])   = 2

No — NOI (Node of Interest) is a special prompt node added to each subgraph at inference time to tell the GNN what task to perform (classify this node? predict this link?). It's not for validation/testing — it's a task-conditioning mechanism present in every forward pass.

NOI is added by the get_edge_list() function in OFA/data/single_graph/gen_data.py. The rules are:

Connections NOI gets:

n2n edges from all feature nodes → NOI (context flows into it)
n2c edges from NOI → class nodes (NOI queries each possible class)
c2n edges from class nodes → NOI (class text flows back)
f2n / n2f edges connecting NOI to the prompt text node (task description)
Rules:

NOI is always a single node per subgraph — the one being queried (target node, head/tail entity, or virtual graph-level node)
It sits between the subgraph (feature nodes) and the class nodes — acting as a bottleneck
For lr_* (few-shot) tasks — no class nodes are attached; NOI just aggregates the subgraph for embedding comparison
For KG tasks (e2e_link) — NOI connects to both head and tail entity nodes, plus 237 relation-type class nodes
For graph-level tasks (chemmol) — NOI connects to all atom nodes in the molecule graph
So the rule is: NOI = target of prediction + bridge to task description + bridge to answer space.

----------------------------------------------------------------------------------------------------------------------------

Important
 NOI aggregates the subgraph into one vector via message passing — after GNN layers, the NOI embedding contains a compressed summary of the entire neighborhood.
Why it doesn't make it too lengthy:

GNN message passing is parallel — all nodes update simultaneously, not sequentially
The subgraph is already small (2-hop neighborhood, capped at ~30 nodes) — not the full graph
NOI just participates in the same message passing rounds as every other node — no extra cost
What NOI ends up containing after GNN:


NOI embedding ← aggregated messages from:
  - feature nodes (local graph structure + text)
  - class nodes (what categories are possible)
  - prompt node (what task to perform)
Then prediction is simply:


similarity(NOI_embedding, class_node_embedding) → highest = predicted class
So NOI is essentially a learned query vector — the GNN writes the answer into it by the final layer. The subgraph is the context, NOI is the question, and its final embedding is the answer.

What the GIN actually does here
The GIN in hard pruning is tiny — 5 layers, 128-dim, ~500K parameters. 
Its only job is to run Deep SVDD: map graphs into a hypersphere, score each graph by distance from center, and rank them. 
It runs once, before training starts. It doesn't need to be smart — it just needs to detect structural redundancy.
